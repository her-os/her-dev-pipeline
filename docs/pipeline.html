/grill — 拷问你的想法 你参与

AI 扮演严格的产品经理，不厌其烦地追问。把你脑子里知道但 AI 不知道的东西全部说出来。

Skill 核心指令（中文翻译）
不厌其烦地拷问我关于这个方案的每一个方面，直到我们达成共识。沿着设计树的每一个分支走下去，逐一解决决策之间的依赖关系。每个问题给出你推荐的答案。

一次只问一个问题，等我回答之后再问下一个。

如果一个问题可以通过探索代码库来回答，那就自己去查代码，不要问我。

当用户使用的术语跟 CONTEXT.md 里的定义冲突时，立刻指出。当用户用模糊的词，提出精确的规范用语。用具体场景来压力测试设计。

AI
邀请码的有效期是多长？永久有效还是有过期时间？
你
7 天过期吧
AI
好的。邀请码用完一次就失效，还是可以多次使用？
你
一次性的
AI
你的 CONTEXT.md 里定义的 "HerClub 会员" 和 "Pro 订阅用户" 是两种身份 — 邀请码是两种用户都能发，还是只有 Pro 用户？
AI
另外我查了代码，user_invite 表已经有 expires_at 字段。我建议复用这张表而不是新建。

原型是用完即丢的代码，用来回答一个问题。问题决定形态：

必要 · /to-prd 你确认

把想法变成需求文档

Skill 核心指令（中文翻译）
把当前对话的上下文和代码库理解合成为 PRD。不要再次拷问用户 — 直接综合已有信息。只做确认性追问。

PRD 质量规则：下一个 Agent 永远无法访问本次对话。必须列出每一个决策、约束、边界条件、被否决的替代方案 — 不允许省略。缺一项就是缺陷。

PRD 共 15 节模板。写入 GitHub Issue → 全新 AI 会话直接读取 → 带着完整上下文开干。

代码地图 + 精准定位

AI 修改代码前，如果不了解代码库全貌，就像闭着眼睛开车。

CodeGraph MCP · 代码地图

类比：GPS 导航

AI 修改代码前，先看清整个项目脉络：每个函数在哪、谁调用谁、改这里会影响哪里。有了地图，AI 修改更准确，同时占用的上下文更少。

使用示例
AI
用 codegraph_context 查看 invite 相关代码结构...
AI
发现 createInvite() 在 3 处被调用，改签名会影响：user-routes.ts:42, admin-api.ts:88, cron-job.ts:15
LSP · 语言服务

类比：代码字典

"这个函数在哪定义？" → 一步跳转。"谁在用它？" → 列出所有引用。没有 LSP，AI 靠搜索猜测，经常猜错。有了它，定位精准、不浪费上下文。

使用示例
AI
用 LSP goToDefinition 跳转到 UserInvite 类型定义...
AI
确认 expires_at 字段类型是 Date | null，可以复用。

上下文管理 — 让 AI 保持聪明

AI 的"注意力窗口"有限。超过 20 万 token 后就会变蠢 — 就像同时处理 50 件事，每件都做不好。

控制在 10 万以内

让 AI 在聪明的范围内做决策。客户端可以看到当前上下文剩余量。

代码地图减少消耗

有了 CodeGraph + LSP，AI 不需要把代码库读一遍。精准定位，省下大量上下文空间。

会话接力 /handoff

快满了 → 压缩为交接文档 → 新会话从断点继续。Bug 调试带 Debug Chain，不走弯路。

/handoff Skill 核心指令（中文翻译）
写一份交接文档，让全新的会话能接续工作。自动获取当前会话 ID，压缩对话为交接文档，引用已有产物而不是复制。Bug 场景包含 Debug Chain：当前阶段 + 反馈环 + 已排除的假设 + 关键发现。

/tdd — 先写测试再写代码

红
先写一个测试（会失败，因为代码没写）
绿
写最少的代码让测试通过
重
优化代码结构，测试仍然通过
Skill 核心指令（中文翻译）
核心原则：测试应该通过公共接口验证行为，而不是验证实现细节。代码可以完全重写，测试不该受影响。

好的测试是集成式的：通过公共 API 走真实代码路径。描述的是系统"做什么"而不是"怎么做"。

规则：一次一个测试，只写刚好够通过的代码。红灯时绝不重构。先变绿再优化。
/tdd 执行流程示例
AI
读取 Issue #42 的 PRD，确认需要实现 3 个行为：创建邀请码、验证邀请码、过期处理。
AI
切片 1/3 — 红：写测试 "创建邀请码返回 8 位字符串，状态为 active"
AI
测试失败（预期）。绿：实现 createInviteCode()，测试通过。
AI
切片 2/3 — 红：写测试 "使用已用过的邀请码应返回错误"
AI
绿：实现验证逻辑。重构：提取 validateInviteCode()。全部测试仍通过。

为什么有用？ AI 不能跳过测试直接交卷。方法论逼迫它用更闭环的方式完成任务。

/e2e-verify — 按验收标准逐条检查

Skill 核心指令（中文翻译）
验证 PRD 验收标准 vs 运行中的代码。只报事实，绝不建议修复。

验证手段优先级（由快到慢）：
1. DB 查询 — 数据状态验证，置信度最高
2. bash / curl / API — 行为验证
3. browser-use — UI 渲染和数据显示验证
4. computer-use — 跨应用 / 非 Web 界面

每条结果：PASS / FAIL / SKIP / BLOCKED。FAIL 一律交 /bugfix。

全 PASS → 提交 PR
FAIL → /bugfix 修复 → 重跑验证（最多 3 轮）
3 轮都 FAIL → 停下来报告人

/e2e-verify 执行示例
你
/e2e-verify #42
AI
读取 Issue #42 验收标准，共 5 条。Round 0/3。
AI
AC-1 创建邀请码 → 查 DB: SELECT * FROM user_invite → PASS
AI
AC-2 过期码拒绝 → curl POST /api/invite/verify → 400 → PASS
AI
AC-4 页面显示 → browser-use 打开邀请页 → 码列表正常渲染 → PASS
AI
AC-3 使用后状态 → 查 DB → status 仍为 active，预期 used → FAIL

/functional-test — 分组手工测试 你参与

分组测试，不要逐个测

以前：注册 → 打勾 → 切账号 → 登录 → 打勾 → 切账号... 很慢。

现在：用一个账号走完整条路线。一组测完再换下一组。

Skill 核心指令（中文翻译）
把散乱的测试点整理成"最短路径测试分组"，然后陪用户跑测试。

技术版（开发者用）— runbook + DB 校验 + 实时反馈
同事版（非技术人员用）— 交互式 HTML + 截图粘贴

只记事实。不分析原因，不建议修复。发现 Bug → 交给 /bugfix。

/bugfix — 7 阶段诊断循环

不是随便改改就交差。像医生问诊一样，系统地排查根因再动手。

Skill 核心指令（中文翻译）
所有 Bug 从这里开始。不允许跳过阶段。

Phase 0 读知识 — 先读 CONTEXT.md + specs + ADR，看有没有已知陷阱
Phase 1 建反馈环 — 建立能快速验证修复的方式（这是整个方法论的灵魂）
Phase 2 复现 — 确认是那个 Bug，确认可复现
Phase 3 假设 — 列 3-5 个排序假设，每个可证伪
Phase 4 探针 — 每个探针对应一个假设，一次一个变量
Phase 5 修复 — 先写回归测试再修
Phase 6 知识沉淀 — 三个问题：学到了什么？有什么坑？架构要改吗？
/bugfix 对话示例
你
邀请码使用后前端状态没更新
AI
进入 Phase 0。读 CONTEXT.md... 读 docs/specs/... 未发现已知陷阱。
AI
Phase 1：建反馈环 — 我会用 browser-use 打开邀请人页面，刷新后观察 status 字段。
AI
Phase 2：复现成功。使用邀请码后，DB status=used，但前端仍显示 active。
AI
Phase 3：假设 1) API 返回了旧数据（缓存）2) 前端没有重新请求 3) WebSocket 推送缺失
AI
Phase 4：探针 — 查 API 响应... 返回 status=used，排除假设 1。检查前端代码... 发现 useEffect 依赖数组缺少 inviteId。
AI
Phase 5：先写回归测试，再修复 useEffect 依赖。测试通过。
AI
Phase 6：知识沉淀 — 写入 docs/specs："useEffect 依赖数组必须包含所有状态变更触发条件。" 完成。

修复后回到 /e2e-verify 或 /functional-test 继续验证。

先合这里测，没问题再合 main。

2.

5.
