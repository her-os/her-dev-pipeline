## 一、当前架构（决策基准线）

### 1.1 服务端组件

| 组件 | 域名 | 部署位置 | 说明 |

**单台服务器，无 CDN，无负载均衡。**

### 1.2 请求链路总览

```
登录/注册：        Salon → POST https://hersoul.cn/api/auth/... → Traefik → Next.js
获取网关配置：     Salon → GET  https://hersoul.cn/api/user/gateway → Next.js 返回 baseUrl + apiKey
AI 对话 SSE：      Salon → Tauri IPC → Rust → Claude CLI → POST https://hersoul.cn/api/v1/messages
自动更新检查：     Salon → GET  https://hersoul.cn/releases/her-beta/latest.json（每10分钟）
安装包下载：       用户 → https://hersoul.cn/releases/her-beta/latest/Her_aarch64.dmg
遥测：            Salon → POST https://hersoul.cn/api/telemetry/...
```

**关键**：AI 对话 baseUrl 不是客户端硬编码的——客户端从 `/api/user/gateway` 获取 baseUrl，服务端下发。默认值 `https://hersoul.cn/api/v1`（生产环境 `API_GATEWAY_CLIENT_BASE_URL` 未设置时）。这意味着服务端可以通过改环境变量切换 AI 对话的域名，客户端重启后生效。

### 1.3 DNS 现状

DNSPod 免费版，共 12 条记录（含 2 条系统 NS）。10 条用户记录需迁移到 EdgeOne DNS。
完整列表见操作手册 §前置准备。

## 二、目标、约束与非目标

### 2.1 核心目标

1. **VPN 用户 SSE 不报错不断连**：开梯子的用户 AI 对话必须能用——不中断、不超时、流式输出流畅
2. **全站加速**：官网、静态资源、API 请求都通过 CDN 边缘节点就近接入
3. **下载加速**：安装包 200MB+，多人同时下载不拖垮源站
4. **AI 对话 SSE 不中断**：首字延迟 >60s 不断连、>10 分钟长对话不中断、流式输出逐字不卡
5. **客户端零改动**：域名不变（hersoul.cn），客户端硬编码的 3 处 URL 无需改动

### 2.2 硬性约束

| 约束 | 说明 |
|------|------|
| 预算 | 不设上限，体验优先 |
| 时间 | 尽快完成 EdgeOne 接入（一周内） |
| 人力 | 一个人执行，需要文档可被其他人/AI 接手 |
| 服务器 | 500 用户前不加第二台服务器 |
| 海外部署 | 暂不部署海外 gateway（模型调用已有中转） |
| 代码改动 | CDN 方案本身不改客户端代码；服务端小改动（SSE 心跳 + 防缓冲头 + Plan B 预埋） |
| VPN 加速 | Plan B（SSE 备用方案）必须保 VPN 用户加速体验，不能绕过加速直连源站 |

### 2.3 非目标（当前阶段不做）与延后事项

| 事项 | 状态 | 什么时候重新评估 |
|------|------|---------------|
| WAF/DDoS 高级防护 | 不做 | 用户量 >5000 或遭受攻击时 |
| 灰度发布/蓝绿部署 | 不做 | 多实例部署后 |
| 多源站负载均衡 | **延后到 500 用户** | 详见§五扩容路线图 |
| WebSocket 支持 | 不需要 | AI 对话用 SSE，不是 WebSocket |
| 多域名拆分 | **延后** | SSE 有问题时启用 stream.hersoul.cn；500+ 用户时考虑 api.hersoul.cn |
| 海外 Gateway | **延后** | 模型中转不够用时，或 VPN 用户延迟仍然不可接受时 |
| HTML 边缘缓存 | 延后到 Phase 4 | 用户量大到 HTML 回源成瓶颈时 |

## 三、决策日志

每个决策格式：**选了什么 → 为什么 → 考虑过什么替代 → 为什么不选 → 什么条件下重新评估**

### D1: CDN 提供商 → 腾讯云 EdgeOne

**选定**：腾讯云 EdgeOne（边缘安全加速平台）

**选择理由**：
- 源站在腾讯云 CVM（广州），同厂商回源走受管网络，延迟最低
- 域名 hersoul.cn 已在腾讯云/DNSPod 管理，生态一致
- EdgeOne 集成 CDN + DNS + 基础安全，不需要分别购买
- 智能加速可优化 VPN 用户的回源链路

**排除的替代方案**：

| 替代方案 | 排除理由 | 什么时候重新考虑 |
|---------|---------|---------------|
| 阿里云 CDN | 跨厂商回源走公网延迟更高；纯 CDN 无安全防护需另购 | 如果迁移整体基础设施到阿里云 |
| Cloudflare Pro | 无国内节点；ICP 备案域名解析到境外 IP 提供服务有合规风险 | 如果只针对海外子域名（如 global.hersoul.cn）且合规确认 |
| 腾讯云 CDN（非 EdgeOne） | EdgeOne 的功能子集，不含 DNS 托管 | 永远不会，EdgeOne 是其升级替代 |
| 不用 CDN，只加带宽 | 不解决地理延迟和 VPN 用户问题；带宽费用随用户增长线性增加 | 永远不会 |

---

### D2: 接入方式 → NS 接入

**选定**：NS 接入 EdgeOne（将 DNS 服务器从 DNSPod 切到 EdgeOne）

**选择理由**：
1. 根域名 `hersoul.cn`（不带 www）不能设 CNAME 记录（RFC1034 规定）。DNSPod 免费版不支持 CNAME 展平
2. 客户端硬编码 `https://hersoul.cn`（3 处），不能只加速 www
3. NS 接入免费，EdgeOne 直管 DNS，根域名加速自然解决

**排除的替代方案**：

| 替代方案 | 排除理由 | 什么时候重新考虑 |
|---------|---------|---------------|
| CNAME 接入（当前状态） | 根域名不能 CNAME；DNSPod 免费版无 CNAME 展平 | 永远不会 |
| 静态子域名 cdn.hersoul.cn | 只加速静态资源，页面/API 不受益 | 永远不会（已选全站加速） |
| 迁移到 www.hersoul.cn | 工程量巨大（cookie、Salon 3处 URL、Traefik、邮件模板全要改），客户端要发版 | 永远不会 |
| DNSPod 企业版 CNAME 展平 | 1800 元/年只买一个 CNAME 展平功能，不提升加速能力 | 如果 EdgeOne DNS 有不可解决的问题需要回退 |

**NS 接入的代价**：
- DNS 传播需 24-48 小时（NS 记录 TTL=86400）
- 传播期间全球结果不一致（部分走 EdgeOne，部分走 DNSPod）——正常现象
- 邮件（MX/SPF/DKIM）必须在 EdgeOne DNS 完整配置，否则传播期间邮件可能异常
- 回滚到 DNSPod 也需要 24-48 小时（Level 1 回滚，最后手段）

### D3: 套餐 → 标准版 3800 元/月

**选定**：EdgeOne 标准版 3800 元/月

**决策变更历史**：v1-v4 选基础版 399 元/月 → v5 改为标准版。

**选择理由**：
- 500 用户一个月内就到，基础版请求数配额（2000 万/月）可能接近上限
- 用户明确表示预算不设上限，体验优先
- 标准版配额更宽裕（流量、请求数、规则数、日志任务数），减少运维关注度
- 标准版 SLA 保障更好

**排除的替代方案**：

| 替代方案 | 排除理由 | 什么时候重新考虑 |
|---------|---------|---------------|
| 个人版 29.9 元/月 | 50GB 流量太小，几百人下载就超 | 永远不会 |
| 基础版 399 元/月 | 原方案选择。500 用户时请求数接近上限，超量费用可能逼近标准版价格 | 如果用户增长放缓到 200 人以下且需要控制成本 |
| 企业版 | 当前不需要企业级 WAF/SLA | 5000+ 用户、需要高级安全防护、或需要明确带宽保障时 |

**重要提醒**：升套餐不等于升性能——EdgeOne 各套餐共用同一网络。标准版主要提升规则、日志、SLA、配额和支持，不直接提升 API 延迟。钱应优先花在路径和容量瓶颈上。

### D4: 智能加速 → 开启

**选定**：开启智能加速（按量付费，0.1 元/万次请求）

**为什么必须开**：
- 静态资源缓存命中后只需「就近接入」（免费）
- 但 API/SSE 每次都回源，VPN 用户体验取决于「回源链路优化」
- 智能加速让边缘节点回源站走 EdgeOne 内部骨干网而非公网
- 源站国际链路差（实测 Google 网络不顺畅），智能加速是 VPN 用户 API/SSE 体验的关键

**费用**：100 用户 ~0.3 元/月，1000 用户 ~3 元/月，1 万用户 ~200 元/月。极便宜。

**不存在替代方案**：这是 CDN 的基础增值功能，关了等于放弃 VPN 用户动态请求加速。

### D5: 回源协议 → HTTPS 不校验源站证书

**选定**：HTTPS 回源（EdgeOne 默认不校验源站证书合法性）

**决策变更历史**：v1 选 HTTP 回源（基于错误前提 "EdgeOne 不支持跳过证书验证"）→ v2 查证官方文档确认默认不校验，改为 HTTPS 回源。

**选择理由**：
1. EdgeOne 默认不校验源站证书（官方文档确认），即使源站证书过期也不导致 525
2. 不需要改 Traefik（消除了 v1 的 Phase -1 整个阶段）
3. 回源链路全程加密
4. 源站 80 端口无需对公网开放

**排除的替代方案**：

| 替代方案 | 排除理由 | 什么时候重新考虑 |
|---------|---------|---------------|
| HTTP 回源 | v1 方案，基于错误前提。存在明文传输风险 | 永远不会 |
| HTTPS + 校验证书 | 证书过期会导致全站 525。当前 Traefik 自动续签，但 NS 切换后续签可能失败 | 设置好自动化证书管理后可考虑 |

**证据**：https://cloud.tencent.com/document/product/1552/125145 原文："默认情况下，EdgeOne 不会对源站证书的有效性进行合法性校验"

### D6: SSL 证书策略

**选定方案**：

| 层级 | 证书 | 管理方式 |
|------|------|---------|
| 边缘（用户→EdgeOne） | EdgeOne 自动签发 Let's Encrypt DV | NS 接入后自动，无需手动 |
| 源站（EdgeOne→CVM） | Traefik 自动签发 Let's Encrypt | 需 acme-challenge-passthrough 规则保持续签 |

**关键风险**：源站证书到期 2026-07-14。NS 切换后 `/.well-known/acme-challenge/` 被 EdgeOne 接管，Traefik HTTP-01 续签可能失败。已加 acme-challenge-passthrough 规则穿透回源解决。

**监控策略**（见操作手册 Phase 1）：
- 2026-06-14：日历提醒检查续签
- 2026-06-30：硬截止线，未续签则手动处理

**手动续签 fallback**：
- 方案 A（推荐，~5分钟）：临时停用 EdgeOne 加速 → Traefik 自动续签 → 重新启用
- 方案 B（无停服）：Traefik 切换 DNS-01 challenge

### D7: SSE 流式透传保障 → 四件套基础配置

**选定**：Phase 0 前全部部署四项防线：

| # | 防线 | 机制 | 状态 |
|---|------|------|------|
| 1 | HTTP 响应超时 600s | EdgeOne 规则 `api-streaming-timeout` | 待配置 |
| 2 | SSE 心跳保活 25s | 代码 `withKeepAlive` 注入 `: keep-alive` | 待部署 |
| 3 | /api/v1/ 关闭压缩 | EdgeOne 规则 `api-v1-no-compression` | 待配置 |
| 4 | 防缓冲响应头 | `no-transform` + `X-Accel-Buffering: no` | 待部署 |

**为什么这四项都是「基础配置」而不是「事后补救」**：

CDN 缓冲 SSE 最常见根因是**响应压缩**。Gzip/Brotli 压缩器需要攒够一个块（4-16KB）才输出，而 SSE 事件通常只有 50-500 字节/条，攒不满 → 表现为「一坨一坨出」。当前 EdgeOne 全局开了 Gzip + Brotli，SSE 响应大概率被压缩缓冲。

所以关压缩不是可选项，是消除最常见根因。四项防线全部就位后，再做 B½ 实测。

**B½ 验证**（Phase 0 阶段 B 和 C 之间的硬性 checkpoint）：
- 四项防线全部就位后，用 `curl --resolve` 强制走 EdgeOne 边缘节点
- 配合 `/api/debug/sse` 测试端点（见 D13），隔离 CDN 问题和模型问题
- **SSE 逐行输出 = 通过 → 继续改 NS**
- **所有行同时输出（被缓冲）= 不通过 → 执行 Plan B**

**决策变更历史**：v1-v3 是「事后三步缓解」→ v4 改为「基础配置四件套」。

### D8: Plan B → GAAP TCP 隧道（stream.hersoul.cn）

**选定**：腾讯云 GAAP（全球应用加速）TCP 隧道

**什么时候触发**：B½ 测试不通过（EdgeOne 仍缓冲 SSE）

**Plan B 架构**：
```
页面/静态/下载:  hersoul.cn         → EdgeOne CDN     → 源站
SSE 对话:        stream.hersoul.cn  → GAAP TCP 隧道   → 源站（Traefik TLS 终止）
```

**为什么选 GAAP**：
- GAAP 是四层 TCP 隧道，不解析 HTTP，物理上不可能缓冲 SSE
- VPN 用户连到最近的腾讯全球入口点，走腾讯内部骨干网，加速效果与 EdgeOne 智能加速同级
- 费用合理：~150-300 元/月（100 用户规模）

**客户端怎么切换**（无需改客户端代码）：
1. 服务端 `.env.production` 设 `SSE_PROXY_BASE_URL=https://stream.hersoul.cn/api/v1`
2. 重新部署 her-web
3. `/api/user/gateway` 开始返回 `stream.hersoul.cn/api/v1` 作为 baseUrl
4. 客户端重启后自动拉取新地址

**客户端更新传播时机**：
- 新登录/重启：即时生效
- 已登录用户遇 AI 对话错误：`InputBar.tsx` 自动重拉
- 手动：用户打开设置页触发同步
- 最坏：重启 Salon 后生效

**回退**：删除 `SSE_PROXY_BASE_URL` → 重新部署 → 回到 EdgeOne CDN 通道。5 分钟完成。

**排除的替代方案**：

| 替代方案 | 排除理由 | 什么时候重新考虑 |
|---------|---------|---------------|
| 直连源站（绕过 CDN） | VPN 用户失去加速——直连广州 CVM 国际链路差，比没有 CDN 更糟 | 永远不会作为 Plan B |
| 独立 API 域名 api.hersoul.cn + 动态加速 | 复杂度更高，需要额外配置 CDN 规则和回源 | 500+ 用户时可考虑，但 GAAP 更简单直接 |
| Cloudflare 作为海外备用 | 合规风险，国内用户可能被导过去 | 永远不会作为 Plan B |

**不触发 Plan B 时**：stream.hersoul.cn 不创建、GAAP 不购买、SSE_PROXY_BASE_URL 不设置。零额外成本。

---

### D9: COS 安装包迁移 → Phase 2

**选定**：Phase 2 执行（NS 切换稳定 48h 后）

**为什么不提前到 Phase 0**：
- 当前用户不从 hersoul.cn 下载安装包（通过其他渠道分发）
- NS 切换是当务之急，同时做 COS 增加复杂度和出错面
- Phase 2 单独做，出问题可以秒级回退（禁用 COS 回源规则即可）

**COS 方案**：
- 桶名：her-releases-1398915892（ap-guangzhou，私有读 + EdgeOne 授权）
- `/releases/her-beta/` 路径回源 COS 而非 CVM
- latest.json 不缓存（每 10 分钟检查更新）
- 版本化文件（DMG/tar.gz）强制缓存 30 天
- 发版时同步上传 COS + 清 CDN 缓存

---

### D10: 域名拆分 → 延后

**选定**：当前不拆分域名。stream.hersoul.cn 仅作为 Plan B 备用。

**当前状态**：所有业务在 `hersoul.cn` 一个域名下，客户端硬编码 3 处。NS 接入后无法按路径绕过 CDN——只能用规则控制不同路径的缓存/超时行为。

**为什么不拆**：
- 100-500 用户，单域名 + 路径规则完全够用
- 拆域名意味着维护多套 CDN 规则、多张证书、多个回源配置
- stream.hersoul.cn 已覆盖最大风险（SSE 缓冲）
- 如果 EdgeOne SSE 没问题，完全不需要拆

**什么时候考虑拆**：
- SSE 在 EdgeOne 上有问题 → 启用 stream.hersoul.cn（Plan B）
- 500-2000 用户 → 考虑 api.hersoul.cn 将 AI API 从主站分离
- API 和静态资源需要完全不同的 CDN 策略时

---

### D11: 多实例部署 → 延后到 500 用户

**选定**：当前单机运行。方案文档里写清楚 500 用户时的扩容步骤（见§五）。

**为什么不现在做**：
- 100-200 用户单机够用
- 加第二台服务器需要 CLB + 多实例部署 + 无状态改造，工作量大
- 当前首要任务是 CDN 接入解决 VPN 用户问题

**什么时候做**：500 用户前启动准备（见§五 扩容路线图）。

**风险**：500 用户如果同时 AI 对话，每条占一个长连接。Node.js（her-web）和 Next.js API route 长时间持有连接需要关注 fd 和内存。

---

### D12: 海外 Gateway → 不做

**选定**：不部署海外 gateway 节点。

**背景**：外部 Review 建议部署香港/新加坡/美西 gateway，理由是「广州源站访问海外模型慢」。

**为什么不做**：
- 国外模型已有搬瓦工中转服务器加速，链路问题已解决
- 国内模型链路正常
- 部署海外 gateway 需要海外服务器 + 跨区域鉴权 + 数据同步，远超当前需求
- EdgeOne 智能加速已优化「用户→源站」链路

**什么时候重新考虑**：
- 中转服务器不够用或不稳定
- VPN 用户延迟在 CDN 接入后仍然不可接受
- 用户规模 >5000 且海外用户占比高

---

### D13: Debug SSE 端点 → 采纳

**选定**：新增 `/api/debug/sse` 测试端点

**来源**：外部 Review 建议

**用途**：
- B½ 验证时用来测试 CDN 是否缓冲 SSE（隔离 CDN 问题和模型问题）
- 上线后排障（用户反馈 SSE 卡顿时，先测 debug 端点判断是 CDN 还是模型的问题）

**行为**：每秒发一条 SSE 事件 `data: {"tick":1,"ts":"2026-05-22T10:00:01Z"}\n\n`，持续 N 秒（query param `?duration=120` 控制，默认 120 秒）。

**代码量**：~20 行，新建 `src/app/api/debug/sse/route.ts`。

---

### D14: SLO 基线 → 轻量采纳

**选定**：采纳轻量版 SLO 作为 B½ 和 Phase 1 的判断标准

**来源**：外部 Review 建议

| 指标 | 目标值 | 测试方法 |
|------|--------|---------|
| SSE debug tick 间隔 | P95 < 1.5s | /api/debug/sse 每秒一条，timestamp 间隔不超过 1.5s |
| SSE 长连接稳定性 | 10 分钟连接 0 断流 | /api/debug/sse?duration=600 |
| SSE 无批量 flush | 逐条输出 | 每条 timestamp 递增，不出现多条 timestamp 相同 |

**为什么是「轻量版」而不是完整 SLO 体系**：100-200 用户不需要 k6 压测和全球拨测。等用户量到 2000+ 再建立正式 SLO。

---

### D15: 加速区域 → 全球

**选定**：EdgeOne 加速区域从「仅中国大陆」改为「全球」

**为什么必须改**：当前设置为「仅中国大陆」，VPN 用户从海外出口的请求不会走海外边缘节点——等于 VPN 用户完全不享受 CDN 加速。这是 Phase 0 的硬性前置条件。

## 八、缓存规则设计

### 8.1 最终规则清单（按优先级排序）

| 序号 | 规则名 | 匹配条件 | 动作 | 说明 |
|------|--------|---------|------|------|
| 0 | acme-challenge-passthrough | URL Path 前缀 `/.well-known/acme-challenge/` | 不缓存，直接回源 | 源站证书续签穿透 CDN |
| 1 | api-streaming-timeout | URL Path 正则 `^/api/v1/` | HTTP 响应超时 600s | SSE 流式响应防断连 |
| 1.5 | api-v1-no-compression | URL Path 正则 `^/api/v1/` | Gzip=OFF, Brotli=OFF | 消除 SSE 缓冲最常见根因 |
| 2 | API-no-cache | URL Path 正则 `^/api/` | 不缓存 | API 请求透传（含 /api/debug/sse） |
| 3 | latest-json-no-cache | URL Path 精确 `^/releases/her-beta/latest\.json$` | 不缓存 | 自动更新检查必须实时 |
| 4 | 大文件缓存30天 | 后缀 mp4 dmg exe gz zip jpg png gif webp ico svg woff2 | 2592000s，强制缓存 | 静态资源 |
| 5 | NextJS静态缓存 | URL Path 正则 `^/_next/static/` | 31536000s，强制缓存 | 有 hash 文件名 |
| 6 | （默认） | 其他 | 遵循源站 Cache-Control | HTML 页面等 |

### 8.2 规则逻辑说明

- **视频 `/her/her-video-web.mp4` 强制缓存 30 天**：源站返回 `max-age=0`，不强制缓存则每次回源。更新视频后需手动清 CDN 缓存
- **`/_next/static/*` 强制缓存 365 天**：文件名含 content hash，安全
- **API 不缓存**：所有 `/api/*` 路径不缓存
- **HTML 页面当前不被 CDN 缓存**：`/zh` 返回 `Cache-Control: private, no-cache, no-store`。CDN 对 HTML 的唯一收益是就近 TLS 终止
