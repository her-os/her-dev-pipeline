### 2026-05-13 her-gateway roome MiMo 400 请求体归因

> 摘要：基于 roome gateway 新增的 `request_log`，确认 MiMo 间歇性 `400 Param Incorrect` 不是模型未配置、额度不足、stream、`max_tokens` 或某个单一渠道不可用，而是 `thinking=adaptive` 与历史 `tool_use/tool_result` 消息结构不兼容：失败请求里的历史 assistant `tool_use` 消息缺少对应 `thinking/redacted_thinking` block。

**证据**：
- request log 开启后，roome MiMo 共 29 条：21 条成功、8 条失败；8 条失败只属于 2 个 request id，每个都被重试 4 次。
- 失败链路均为同一用户/token，渠道从 `8 -> 9 -> 9 -> 9`，前后相邻请求可成功，排除“模型不存在/渠道整体不可用”。
- 失败请求体共同特征：顶层 `thinking={"type":"adaptive"}`、保留 tools、历史里有 `assistant tool_use` 与后续 `user tool_result`，但该 assistant 消息没有 `thinking/redacted_thinking` block。
- 对失败请求体做回放：
  - 原始请求、非 stream、降低 `max_tokens` 仍返回 400。
  - 显式改成 `thinking={"type":"disabled"}` 后返回 200。
  - 删除那段历史 tool 调用交换后返回 200。
  - 删除 `output_config`、删除 system cache-control、仅去掉顶层 thinking 都不能修复。

**判断**：
- 前端文案 `There's an issue with the selected model ...` 是客户端把 400 归类成“所选模型不可用”的提示，gateway 日志里没有这个原文。
- 当前 roome MiMo 渠道里的“400 retry 时剪历史 thinking/redacted_thinking”规则对这类请求是反方向的；tool history 场景可能需要保留 thinking block，剪掉会让重试继续失败。

**建议修复**：
- gateway 侧先加 MiMo 专用兜底：最终 Claude 请求体如果同时满足 `thinking=adaptive/enabled`、历史里存在 `tool_use/tool_result`、对应 assistant tool_use 消息缺少 thinking block，则把本次请求顶层 thinking 显式降为 `disabled`，保住对话连续性。
- 同步移除或收窄 MiMo 渠道里 retry 时剪 `thinking/redacted_thinking` 的 param_override。
- 对这类 `400 Param Incorrect` 避免无意义跨渠道重复重试，除非 retry 前已经改写请求体。
- 长期应在客户端或 Claude-format 转换层保留带 tool_use 的 assistant thinking/signature 历史，避免 gateway 只能降级思考参数。

- `{"*": ""}` 是单渠道快速止血方案，会跳过 Authorization/Cookie/Host 等危险头；这次先补规则白名单，影响面更小。
- 负向样本没有稳定复现原始 `Invalid signature`：有效链路已经覆盖真实 Claude Code header、prompt cache beta、thinking signature、多轮回传；但不能据此承诺所有非法 signature 形态都会被这个 header 修掉。
